# 変更履歴

このファイルには、プロジェクトのすべての重要な変更が記録されます。

## [v2.3.0] - 2026-05-23

集計起算日カスタマイズを目玉機能とするマイナーリリース。月次・年次集計のサイクル境界をユーザーが自由に設定できるようになり、給料日サイクルや年金日サイクルで家計を集計可能になりました。あわせて全集計画面・ユーザ管理画面のウィンドウレイアウト改善、ダッシュボードの基準日表示、Linux fcitx5 環境でのログイン IME 制御も同梱しています。

### 新機能

- **集計起算日カスタマイズ** (#24): 月次起算日 (1〜31) / 年次起算月 (1〜12) + 年次起算日 (1〜31) をユーザー単位で設定可能に。給料日 25 日や年金振込日 15 日などに合わせて家計集計サイクルを構築できます。起算日が存在しない月（例: 2 月の 30 日）は末日で自動代用。ダッシュボードの月期ラベルは始点月で呼ぶ形（例: `2026年5月（5/13〜6/12）`）に変更
- **ウィンドウ自動フィット + 中央配置**: ユーザ管理・全集計画面で起動時にウィンドウ高さを画面サイズに合わせ、現在のモニタで中央配置。集計結果テーブルは固定高さで内部スクロール、メインコンテンツ全体は外側でスクロール可能に
- **ダッシュボード残高の基準日表示**: 口座別残高の表記を `2026-05-31 時点` / `as of 2026-05-31` に改善

### 改善

- **ログイン画面の IME 制御 (Linux)**: ibus に加えて fcitx5 環境でもユーザー名フィールドの focus 時に自動で直接入力モードへ切り替え。`fcitx5-remote -c` を Tauri command 経由で呼び出し（fcitx5 入力メソッドリスト先頭が英語キーボードであることが前提）

### バックエンド改修

- `services/aggregation.rs` の `YearStart` enum を廃止し、`monthly_aggregation` / `yearly_aggregation` の境界計算に `start_day` / `(start_month, start_day)` を直接受け取る形に変更
- 純粋関数モジュール `services/period.rs` を新設、`recurring.rs` と重複していた `end_of_month` 実装を統合
- 集計画面の Tauri command (`get_monthly_aggregation` / `get_yearly_aggregation` 他) がユーザー設定から起算日を自動取得

### i18n

- 26 リソース追加 (RESOURCE_ID 2323〜2348) — Period Settings UI 関連の `user_mgmt.*` キー 20 件、`validation.invalid_period_start_*` 4 件、`dashboard.balance_as_of` 2 件

### テスト

- Rust: 350 → 371 件（period.rs 16 件 + aggregation.rs 7 件 + validation.rs 8 件追加）
- JavaScript (Jest): 623 件全 PASS

### 既知の制約

- Linux + fcitx5 で完全な IME OFF を得るには、`fcitx5-configtool` で入力メソッドリストの先頭に英語キーボードを設定する必要があります
- 起算日が土日祝に当たった場合のサイクル境界シフト（前/後営業日への自動調整）は v2.4.0 以降の対応とします (#57)

## [v2.2.0] - 2026-05-19

トースト通知・バリデーション基盤・alert→toast 移行を一括適用するマイナーリリース。`alert()` の呼び出しを全 20+ 箇所でノンブロッキングな toast に置き換え、フォームの入力制限をリアルタイムで表示するバリデーション基盤を全管理画面に展開した。

### 新機能

- **トースト通知コンポーネント** (#45): ノンブロッキングな成功・エラー・情報メッセージを画面右上に表示する `showToast()` ユーティリティを導入。`alert()` の代替として設計し、画面操作をブロックしない UX を実現
- **統一バリデーション基盤** (#37): 文字数上限のあるすべての入力フィールドに「現在文字数/上限」カウンターをリアルタイム表示するバリデーション基盤を追加。上限超過時は赤色表示でユーザーに即時フィードバックを提供
- **全管理画面への alert → toast 移行** (#50): アカウント管理・取引管理・カテゴリ管理・店舗管理・メーカー管理・商品管理など全画面の `alert()` 呼び出し 20+ 箇所を `showToast()` に置き換え。ナビゲーション連動のアクセス拒否（画面遷移時）のみ `alert()` を維持（意図的残留）

### 改善

- **ログイン画面のオートフォーカス**: ログイン画面へ遷移するたびにユーザー名フィールドが自動フォーカスされ、即座にキーボード入力が可能に
- **ヘッダー合計再計算ダイアログのデフォルト変更**: 明細変更後の「ヘッダー合計を再計算しますか？」ダイアログのデフォルト選択を「再計算する」から「現在の合計を維持」に変更。意図せず合計が書き換わる誤操作を防止

### i18n

- アカウント管理・取引管理・カテゴリ管理の toast エラーメッセージ用に 11 キー（en/ja 計 22 リソース）を追加。RESOURCE_ID 2301–2322 を使用

### テスト

- 全体: 800 → 973 テスト（Rust: 201 → 350, JavaScript: 599 → 623）

## [v2.1.0] - 2026-05-04

繰り返し予定入出金機能を追加するマイナーリリース。月給・公共料金・サブスクなどの定期取引を「ルール 1 件登録 → 期間内の予定取引を一括生成」できるようになった。

### 新機能

- **繰り返し予定入出金ルール**: `Admin > Recurring Rule` 画面で、周期・期間・テンプレート（金額・カテゴリ・口座・メモ）を 1 件のルールとして登録すると、期間内の `IS_SCHEDULED=1` 予定取引が一括生成される。生成された各予定取引はルール由来であることを `RULE_ID` で記録するが、テンプレート（ルール）と実データ（生成済み予定取引）は独立した存在として扱う設計
- **対応サイクル（v2.1.0 UI から作成可能）**: 日次（n 日毎、起点日指定）／月次（n 月毎、日付指定）／月次（n 月毎、第N週指定曜日）。Weekly / Yearly / 月末・月末代用モードはバックエンド全対応済みだが UI は v2.1.x 以降
- **第N週指定曜日**: 例「毎月第4木曜日」のような指定が可能。Zaim 等の競合では未対応の差別化要因（KakeiBon を作った主要動機の 1 つ）。第5週指定 = その月の最終週として扱う
- **土日祝シフト**: 生成日が土日祝に当たった場合、「前倒し（直前の平日）」「後倒し（直後の平日）」「変更なし」の 3 モードから選択可能。給料日（前倒し）／引落日（後倒し）の業務慣行に対応。連続休日（GW・年末年始）も平日まで遡る／進む
- **日本祝日マスタの自動生成**: `jpholiday` クレートを依存に追加し、起動時に「当年 -5 / +10 年」の 16 年分を `HOLIDAYS_STANDARD` に `INSERT OR IGNORE`。年や法改正に対するコード変更不要
- **ルール一覧 UI**: 登録済みルールの確認、件数表示、削除ボタン
- **削除モードの 2 択**: 「ルールのみ削除（生成された予定取引は単独取引として残す）」「ルールと生成済み予定取引を全て削除」をモーダルで明示的に選択

### 修正・改善

- **`PRAGMA foreign_keys = ON` を本番接続で有効化**: KakeiBon の SQLite 接続では従来 OFF（SQLite 既定値）のままだったため、宣言済みの `ON DELETE CASCADE` / `SET NULL` が全て無視されていた。今回の削除機能実装で発見・修正。今後は `TRANSACTIONS_DETAIL` などの既存 CASCADE も実際に効く
- **既存単発予定取引のマイグレーション**: v2.0.x で登録済みの `IS_SCHEDULED=1` 単発取引は、新規 `RULE_ID` カラムが NULL のまま自動扱い（ルール由来でない単発として正常認識される）

### スキーマ変更

- `TRANSACTIONS_HEADER` に `RULE_ID INTEGER`（FK ON DELETE SET NULL、新規 DB のみ宣言。既存 DB はアプリ層で整合性確保）
- `USERS` に `HOLIDAY_LOCALE TEXT DEFAULT 'JP'`、`WEEK_START_DAY INTEGER DEFAULT 1`
- 新規テーブル: `RECURRING_RULES`、`RECURRING_RULE_DETAILS`（`RULE_ID UNIQUE` で 1:1 制約）、`HOLIDAYS_STANDARD`、`HOLIDAYS_USER_CUSTOM`

### テスト

- `services::recurring` に 45 ケースのテスト追加（純粋関数 30 + ラウンドトリップ 8 + 検証エラー 6 + ISO 曜日マッピング 1）

### v2.1.x / v2.2.0 で対応予定

- Weekly / Yearly / 月末モード（DayOfMonthOrEnd / EndOfMonth）の UI 入力（バックエンドは対応済み）
- ルール編集機能（現状は「削除→再作成」で代替可能）
- 週始まり日設定 UI（Weekly UI と一緒に）
- 祝日カスタマイズ UI（HOLIDAYS_USER_CUSTOM）

## [v2.0.1] - 2026-05-01

v2.0.0 リリース後に発見された i18n 関連バグ・テスト基盤の不具合を修正したパッチリリース。

### 修正

- **設定テストのフラキー解消** (#23, PR #27): `settings::tests::test_save_and_reload` が並列実行時に約 5 回に 1 回失敗していた問題を修正。`env::set_var("HOME", ...)` がプロセス全体で共有されており、別テストの cleanup で同一の temp ディレクトリが消されていたのが原因。`tempfile` クレートを dev-dependencies に追加し、`SettingsManager::with_path()` を新設、各テストが独立した `TempDir` を持つ構造に書き換えた。`cargo test --release` 5 連続実行で 266/266 pass を確認
- **EN モードで Font Size メニューが日本語のまま** (#25, PR #28): 言語切替後にトップメニューの Font Size サブ項目（Small/Medium/Large/Custom 相当）が翻訳されない問題を修正。`menu.js` の `handleLanguageChange()` に `setupFontSizeMenu()` の再描画呼び出しを追加。Font Size メニュー項目は `data-i18n` 属性なしで動的生成されているため `i18n.updateUI()` の対象外だった
- **Font Size メニュー再描画漏れを 8 画面に展開** (#31, PR #32): #25 では `menu.js` のみ修正されていたが、ダッシュボード・各管理画面・集計画面など 8 画面にも独自の `handleLanguageChange` / `changeLanguage` 関数があり、同じ漏れが残っていた。各画面に 1 行ずつ `await setupFontSizeMenu();` を追加
- **Language メニューの表示が画面ごとに不統一** (#26, PR #29): 9 画面のうち 3 つの異なる表示パターンが混在していた問題（`English / 日本語` mix、`English / Japanese` localized、`en / ja` raw code）。バックエンド `get_language_names` を「言語コード自身のロケール」で resource 引きするネイティブ表記モードに変更し、フロント 4 ファイル（menu.js, dashboard.js, category-management.js, aggregation.js）のハードコード/バグを修正。GitHub や Wikipedia と同じ言語切替慣習に統一。`category-management.js` は `Vec<(String,String)>` をオブジェクトとして読んでいた実装ミスがあり、`aggregation.js` はタプル配列に対して `Object.entries` を使っていた

### ドキュメント

- README に Screenshots セクションを追加（ダッシュボード・入出金編集・明細編集・ユーザー管理の 4 枚）

## [v2.0.0] - 2026-05-01

税計算ロジックの全面刷新を含むメジャーリリース。明細単位で税計算→丸めを行ってから集計していた従来方式では、丸め誤差が累積して集計値がレシートの実額と一致しないケースがあった（4月の食費集計で 107 円差が判明したのが発端）。本リリースで、計算式を「1 取引内では税率ごとに SUM(税抜) → 税計算 → ヘッダーの TAX_ROUNDING_TYPE で丸め → 税率合算」「取引をまたぐ集計では既に整数化された値をそのまま SUM、追加丸めしない」に統一した。

### 破壊的変更

- **AMOUNT のセマンティクス**: 取引明細の `AMOUNT` は税抜額固定として扱う。`AMOUNT_INCLUDING_TAX` は UI / DB ともに残し、税込表示用の派生値とする
- **集計クエリの全面書き換え**: 明細レベルのサブクエリで税込/税抜を判定し、ヘッダー単位で丸めた値を集計する方式へ。重複計上バグと累積誤差を解消（commit `dc720521`）

### 新機能

- **ヘッダー TOTAL_AMOUNT の自動再計算**（commits `984987d5` `d600a2a4`）: 明細編集・ヘッダー編集後にプロンプトを表示し、「OK で自動値で上書き／キャンセルで手動値維持」の選択を提供
- **税方式変更時のリアルタイムプレビュー**（commit `31cadb09`）: ヘッダー編集モーダルで税方式（税抜/税込/丸め方式）を変更すると、合計欄が即座に再計算されて表示される（フロント JS 純関数 + Jest テスト）
- **一括再計算 + ワンクリックロールバック**（commits `83d90e78` `f7d61a13`）: ダッシュボードに「データメンテナンス」セクションを追加。実行前に DB ファイルを自動バックアップし、不本意な結果になった場合は 1 クリックで元の状態に戻せる
- **パターンマッチ式 recalc**（commit `b6cbab91`）: ユーザーが入力した `TOTAL_AMOUNT` を正として、4 パターン（税抜 floor / half-up / ceil + 税込）を総当たりで検証。一致するパターンが見つかれば `TAX_ROUNDING_TYPE` / `TAX_INCLUDED_TYPE` を補正、一致しない場合のみ `TOTAL_AMOUNT` を上書き。詳細な変更ログを表示
- **口座別残高スナップショット**（commit `f888c8bc`）: ダッシュボード上部に口座ごとの残高パネルを追加。元データとの突き合わせ検証に使える

### 修正

- **ceil 丸めで 1 円欠落**（commit `7bd04550`）: SQLite の `CAST(-x/100 AS INTEGER)` イディオムが trunc-toward-zero の影響で実際は floor になっていた。`(x + 99) / 100` への書き換えで ceil として正しく動作するよう修正。Rust 側 / JS 側は最初から正しかったので影響範囲は SQL のみ
- **SQLite ATTACH / UPDATE / DETACH の同一接続化**（commit `7d844f7c`）: 一括再計算で別接続が DETACH を見られず NOT FOUND になる不具合を解消
- **ダッシュボードカラムのスクロール**（commit `6e27e3f2`）: コンテンツがビューポートを超えるときにカラム内でスクロールできるよう修正

### 開発体験 / メンテナンス

- **@tauri-apps/cli を ^2.11.0 に更新**（commit `d354b689`）

### バージョン方針

`AMOUNT` のセマンティクス変更は API レベルの破壊的変更に相当するため MAJOR を上げて v2.0.0 とした。データベースのスキーマ変更はないので、既存ユーザーは特別な移行作業なしにアップデート可能。集計値が以前と異なる場合があるが、これは丸め誤差の修正によるもので、新しい値の方がレシート実額と一致する。
